Ingestion & Chunking

Ingestion Pipeline & Contextual Chunking

The ingestion module processes multi-format documents (PDF, DOCX, MD, and code scripts), normalizes the content, splits it into digestible chunks, and embeds them into the database.

Multi-Format File Loaders

The ingestion system supports recursive directory crawling and handles files according to their file extensions:

  • PDF Files: Handled by pdfplumber to extract layout-aware text along with page metadata.
  • DOCX Files: Parsed using python-docx to extract paragraph text, headings, and tabular cells.
  • Markdown: Scrapes structural indicators (headers, blockquotes, raw links).
  • Code Files: Handles developer code files (e.g. .py, .js, .ts, .java) with specific code split rules.

Scanned PDF Support (ONNX OCR Fallback)

Standard loaders extract blank text if a PDF contains only scanned images (e.g., scanned resumes or scanned forms). To solve this, our loader integrates a **local OCR fallback pipeline** using rapidocr-onnxruntime:

OCR Fallback TriggerIf pdfplumber returns 0 text extraction characters, the loader renders each page to an image and runs a local RapidOCR ONNX model to extract characters, ensuring zero-configuration OCR capability.

Chunking Strategies

RAG performance relies heavily on chunk boundaries. We implement 5 strategies in src/ingestion/chunker.py:

1. Fixed-Size Chunking

Splits text into fixed sizes (e.g. 512 tokens) with a sliding overlap (e.g. 50 tokens). Fast but frequently cuts off sentences in the middle.

2. Recursive Text Splitting

Splits text recursively using delimiters (paragraphs \n\n, lines \n, sentences, and words) until size constraints are satisfied, maintaining paragraph integrity.

3. Markdown Chunking

Splits content according to markdown headers (#, ##, ###), grouping topics together.

4. Semantic Chunking

Computes embeddings for each sentence, calculates similarity between adjacent sentences, and places chunk boundaries where cosine similarity drops below a statistical threshold.

5. Contextual Chunking (SOTA)

Prepends a global context summary generated by the LLM to each individual chunk content before indexing, mitigating context loss during vector retrieval.

State-of-the-Art Contextual Chunking

During ingestion, the chunker reads the first 4,000 characters of a document and triggers a fast local LLM completion to construct a document context summary. For each chunk, it wraps the content with this context summary:

# Context Injection Template
CONTEXT_TEMPLATE = """
You are a context injector. Given a document and a small chunk of text from it, write a short (1-2 sentences) context statement that situates this chunk within the overall document.
Do not summarize the chunk itself.

Overall Document:
{document_context}

Chunk to situate:
{chunk_content}

Output only the 1-2 sentence context statement, nothing else.
"""

# Resulting Chunk Indexed:
# <Document Context: [Summary]> [Actual original chunk text content]
Result: By prepending the context summary, standard dense embeddings preserve crucial identifiers (like project names, document subjects, and core authors) that would otherwise be missing from a tiny text block.

Configuring Chunking

Open config.yaml to choose your strategy:

chunking:
  strategy: "contextual" # Options: fixed, recursive, semantic, markdown, contextual
  chunk_size: 512
  chunk_overlap: 50
  semantic_threshold: 0.82
